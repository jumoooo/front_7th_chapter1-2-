# AI와 테스트를 활용한 안정적인 기능 개발 리포트

## 사용하는 도구를 선택한 이유가 있을까요? 각 도구의 특징에 대해 조사해본적이 있나요?

- Cursor Pro + Claude Sonnet 4.5
  사용 목적: AI Agent로 활용하여 실제 코드 작성, 테스트 수행, 산출물 생성 등을 담당하였다.
  특징: 전반적으로 코드 개발에 최적화되어 있으며, 프롬프트에 명시된 규칙을 충실히 이행하는 안정적인 특성을 보였다. 다만, 문장 작성이나 서술형 표현 능력은 GPT 시리즈에 비해 다소 약한 편이었다.
- Gemini CLI 2.5 Pro
  사용 목적: 프로젝트 단위로 AI Agent, Rule, 실행 프롬프트의 초안(초판본)을 작성할 때 주로 사용하였다.
  특징: 명시적인 제약이 없는 한 긍정적 방향으로 일관된 답변을 제공하는 보수적 성향을 보였다. 또한 Pro 버전은 50회까지 무료 요청(Request)을 지원한다는 장점이 있다.
- GPT-5
  사용 목적: Gemini CLI가 생성한 문서를 구조적으로 다듬고, 체계에 맞게 보완하는 용도로 사용하였다. 또한 Cursor Pro가 비정상적인 결과를 도출할 경우 논의 및 검증 단계에서도 활용하였다.
  특징: 텍스트 기반 문장 완성 능력이 탁월하며, 전반적으로 균형 잡힌 다목적 언어 모델로 평가된다.

- 실제 경험 중 하나로 ‘Cursor Pro + Claude Sonnet 4.5’ 에게 “매월 31일마다 실행되는 요청 문서” 를 작성해 달라고 한 적이 있었습니다.
  그런데 결과물에서 “31일만”이라는 표현이 여러 가지로 해석될 수 있는 애매한 문장을 반환했습니다.
  같은 요청을 GPT-5에게 전달했을 때는 상황이 달랐습니다. 문맥에 따른 의미를 명확히 구분하고, 다른 해석의 여지가 없는 깔끔하고 일관된 문장으로 완성해 주었습니다.
  이 경험을 통해 각 AI 모델이 언어 해석의 정밀도나 문맥 처리 능력에서 차이를 보인다는 점을 체감할 수 있었고, 작업의 성격에 따라 적절한 모델을 선택하는 것의 중요성을 다시 한 번 느꼈습니다.

## 테스트를 기반으로 하는 AI를 통한 기능 개발과 없을 때의 기능개발은 차이가 있었나요?

- 테스트 기반 미적용 시
  기능 요청의 의도는 일정 부분 파악할 수 있었지만, 완성된 결과물에는 오류가 다수 발생하여 후속 수정이 자주 필요했습니다. 결과적으로 개발 효율성과 코드 품질이 모두 낮게 유지되는 경향이 있었습니다.
- 테스트 기반 적용 시
  AI Agent가 개발 과정 중에 버그를 조기에 탐지할 수 있었으며, 테스트 결과를 기반으로 산출물을 검토함으로써 코드의 신뢰도와 안정성이 향상되었습니다. 또한 테스트 코드를 중심으로 개발이 진행되면서 코드 전반에 일관성이 확보되는 효과가 있었습니다.

## AI의 응답을 개선하기 위해 추가했던 여러 정보(context)는 무엇인가요?

- 프로젝트의 구조, 의존성, 핵심 설정 파일 등을 분석하도록 특정 AI Agent에 지시하여, 이를 기반으로 산출물을 자동 생성하도록 하였습니다.
- 산출물의 일관성과 품질 유지를 위해 각 AI Agent별 산출물 템플릿을 사전에 정의하고, 생성 시 해당 템플릿을 참조하도록 명확히 지정하였습니다.

## 이 context를 잘 활용하게 하기 위해 했던 노력이 있나요?

- 다른 AI Agent가 동일한 문서를 참조하도록 하여, 프로젝트 구조 및 경로 정보에 대한 인식이 일관되도록 통일화를 진행하였습니다.
- 컨텍스트, 프롬프트, 그리고 AI Agent 간의 우선순위를 명확히 설정함으로써, 작업 흐름의 충돌을 최소화하고 보다 효율적인 협업 환경을 구축하였습니다.

## 생성된 여러 결과는 만족스러웠나요? AI의 응답을 어떤 기준을 갖고 '평가(evaluation)'했나요?

- 전반적으로 생성된 결과물의 완성도는 양호한 수준이었습니다.
- 평가 기준은 다음과 같이 설정하였습니다
  - 기능 요청 문서, AI가 작성한 초기 기획 산출물, 실제 테스트 시 구현된 기능 간의 일치 여부
  - 내부 AI Agent별 평가 점수를 종합하여 총 점수가 90% 이상인지 여부
  - 테스트 코드의 통과율이 100%에 도달하는지 여부

## AI에게 어떻게 질문하는것이 더 나은 결과를 얻을 수 있었나요? 시도했던 여러 경험을 알려주세요.

- 우선순위를 지정해 주기
  AI가 판단을 내릴 때 기준이 명확해져, 핵심적인 부분부터 처리하도록 유도할 수 있었습니다.

- 목표를 직접적으로 제시하기
  문장이 다소 어색하더라도 원하는 결과나 목적을 명확히 제시하면 AI가 의도를 정확히 파악하고 필요한 경우 되물어 올바른 방향으로 조정할 수 있었다.

- 기존 코드 기반으로 추가 코드를 작성하도록 유도하기
  이전에 작성된 코드를 참조해 이어서 개발을 진행하면, 불필요한 재작성이나 구조 파악 과정이 크게 줄어들었습니다.
  또한 프로젝트의 전반적인 흐름과 데이터 구조를 AI가 보다 명확하게 이해할 수 있어, 결과물의 일관성과 완성도가 높아졌습니다.

- 중요한 컨텍스트를 프롬프트에 명시하기
  주요 전제나 배경 정보를 입력 프롬프트에 명확히 포함하면, AI가 맥락을 정확히 이해하고 보다 일관된 방향으로 응답을 생성할 수 있었습습다.

- 파일을 여러개 읽고 피드백 해야하는 작업에서는 파일을 일정 단위로 쪼개서 분석하라고 명령을 내렸습니다.

## AI에게 지시하는 작업의 범위를 어떻게 잡았나요? 범위를 좁게, 넓게 해보고 결과를 적어주세요. 그리고 내가 생각하는 적절한 단위를 말해보세요.

- 각 AI Agent의 작업 정밀도를 최대화하기 위해, 범위를 가능한 한 세부적으로 나누어 주었습니다.
  - 좁은 범위를 여러 AI Agent에 할당하고, 서로 견제하도록 설계하였습니다.
  - 예를 들어, Dev Agent가 작업을 완료하면 QA Agent가 이를 검증하고, 문제가 발견되면 QA가 Dev를 호출하여 재작업하도록 구현했습니다.
  - 이러한 방식으로 AI Agent 간 상호 검증과 피드백 루프를 형성함으로써, 전체 프로젝트의 정확도와 안정성을 높일 수 있었습니다.

- AI Rule(.cursorrules) : "헌법(Constitution)" 입니다.
  - 역할: 프로젝트 전반에 걸쳐 '무엇을(What)' 해야 하는지에 대한 정적이고 근본적인 원칙을 정의
- prompt (프롬프트) : "작전 지시서(Mission Briefing)" 또는 "실행 계획(Execution Plan)"입니다.
  - 역할: 이번에 수행할 특정 작업에 대해 '어떻게(How)' 진행할지에 대한 동적이고 순차적인 절차를 지시합니다.

## AI가 잘하는 것과 못하는 것에 대해 고민한 적이 있나요? 내가 생각하는 지점에 대해 작성해주세요.

- AI가 잘하는 것
  - 일반적인 사람보다 뛰어난 연산 속도를 갖고 있다.
  - 문서 작성 속도와 품질이 높아, 반복적이고 양적인 작업에서 효율적이다.
- AI가 어려워하는 것
  - 작업물의 균일성을 유지하는 데 한계가 있다.
  - 장시간 또는 길어진 작업에서는 명확한 기억 능력이 떨어져, 이전 맥락을 완벽히 유지하기 어렵다.

## 마지막으로 느낀점에 대해 적어주세요!

- 이번 AI 를 사용하는 것은 마치 카페에서 좋은 커피 머신을 쓰는것과 같다고 느꼈습니다.
  좋은 커피 머신은 좋은 커피를 만들어주지 않습니다. 환경에 변화에 일정한 커피의 맛을 책임질 뿐이죠.
  커피의 맛의 퀄리티는 결국 바리스타에게 달려있습니다.
  이처럼 개발의 좋은 퀄리티는 AI 를 사용하더라도 결국에는 사용자의 퀄리티에 따라 결과가 달라진다고 느꼈습니다.

  초보적인 수준의 작업은 AI를 통해 어느 정도 보완할 수 있지만, 중급 이상의 결과물을 안정적으로 만들기 위해서는 개발자의 전문성과 경험이 필수적인거 같습니다.
  이번 과제를 통해 AI를 더 잘 다루는 것보다, 개발 기본기를 탄탄히 다지는 것이 중요하다는 점을 다시 한 번 느꼈습니다.
